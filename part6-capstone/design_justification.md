## Storage Systems

In this architecture, multiple storage systems are selected based on the nature of data and use cases. An OLTP database is used to store structured patient records such as EHR data, ensuring consistency, reliability, and fast transactional queries required for daily hospital operations. For handling large volumes of raw and semi-structured data such as ICU device streams and logs, a Data Lake is used. It allows flexible storage and supports both batch and real-time ingestion without strict schema requirements. Additionally, a Data Warehouse is used to store cleaned and transformed data for analytical purposes like generating monthly reports, tracking bed occupancy, and department-wise cost analysis. For enabling semantic search and natural language queries by doctors, a Vector Database is integrated. It stores embeddings of patient records, allowing efficient similarity search and intelligent querying.

## OLTP vs OLAP Boundary

The system clearly separates transactional and analytical workloads. The OLTP layer handles real-time operations such as updating patient records, doctor interactions, and storing incoming data from devices. It ensures ACID properties and low latency for operational tasks. On the other hand, the OLAP layer begins after data is processed through ETL pipelines. Data is extracted from OLTP systems and the Data Lake, transformed into structured formats, and loaded into the Data Warehouse. This warehouse is optimized for read-heavy analytical queries, such as trend analysis and reporting. By maintaining this boundary, the system avoids performance bottlenecks and ensures that operational queries do not interfere with analytical workloads.

## Trade-offs

One major trade-off in this design is the complexity introduced by maintaining multiple storage systems. While using OLTP, Data Lake, Data Warehouse, and Vector Database improves scalability and performance for specific use cases, it also increases system maintenance, data synchronization challenges, and operational overhead. To mitigate this, a well-defined ETL pipeline is implemented to ensure consistent data flow and transformation across systems. Additionally, orchestration tools and automated workflows can be used to manage data pipelines efficiently. Another trade-off is latency between real-time data ingestion and availability in analytical systems, which can be minimized using near real-time streaming and incremental data updates.
